## Part 2：语言模型到底接收什么

### 从概率问题开始

语言模型对 token 序列分配概率。对序列 $x_1, x_2, \dots, x_T$，自回归分解是：

$$
P(x_1, \dots, x_T) = \prod_{t=1}^{T} P(x_t \mid x_1, \dots, x_{t-1})
$$

模型每一步接收前面的 token，预测下一个 token 的概率分布。问题是：用户输入的是字符串，而神经网络需要有限范围内的整数 ID。

```text
"Hello, 🌍! 你好!"
        │ encode
        ▼
[13225, 11,  ...]  ← token IDs
        │ model
        ▼
下一个 token 的概率分布
```

### 四个容易混淆的层次

| 层次 | 示例 | 说明 |
| --- | --- | --- |
| Unicode string | `"你好"` | 程序员看到的文本抽象，例如 Python `str` |
| UTF-8 bytes | `e4 bd a0 e5 a5 bd` | string 的一种二进制编码 |
| Token | `你`、`你好` 或某段 bytes | tokenizer 选择的处理单元 |
| Token ID | `12345` | token 在有限词表中的整数键 |

<Term term="tokenizer" slug="tokenizer" definition="负责 string 与 token ID 序列之间编码、解码的规则、词表与实现。">Tokenizer</Term> 的最小接口可以写成：

```python
from abc import ABC, abstractmethod

class Tokenizer(ABC):
    @abstractmethod
    def encode(self, text: str) -> list[int]:
        """string → token IDs"""

    @abstractmethod
    def decode(self, ids: list[int]) -> str:
        """token IDs → string"""
```

第一个核心不变量是 round trip：

```python
assert tokenizer.decode(tokenizer.encode(text)) == text
```

这表示编码不会丢失信息。它不表示切分一定合理，也不表示实现足够快。

### 真实 tokenizer 的边界经常违反直觉

官方 trace 使用 tiktoken 展示了几个现象：

- 前导空格可能和后面的单词属于同一个 token，例如 `" world"`。
- 同一个单词位于开头和句中时，token ID 可能不同。
- 长数字通常按若干位一组切分，不一定逐位，也不一定作为完整数字。
- emoji 和中文可能由一个或多个 token 表示。

:::caution[Token 不是单词]
不要用英文单词数估算 token 数。token 边界由 tokenizer 的训练数据、pre-tokenization 规则、merge 表和 special token 共同决定。
:::

## Part 3：如何评价 tokenizer

一个 tokenizer 至少要同时考虑四项指标。

### 1. 可逆性

正常文本应满足 `decode(encode(text)) == text`。如果系统还需要处理任意 byte 序列，则必须进一步明确“非法 UTF-8”如何处理；不是所有 byte 序列都能直接解码成 Unicode string。

### 2. 词表大小

<Term term="vocabulary" slug="vocabulary" definition="tokenizer 可以输出的全部 token ID，以及每个 ID 对应的 byte 序列。">Vocabulary</Term> 大小记作 $V$。更大的词表通常能用更少 token 表示文本，但也会扩大模型的 embedding 和输出层。

若模型维度为 $D$，单个 embedding 表大约含 $V \times D$ 个参数。于是 $V$ 不是“越大越好”。

### 3. 序列长度

序列长度记作 $T$。传统 full attention 的 attention score 数量随 $T^2$ 增长，因此把同一段文本编码得更短通常很有价值。

:::note[不要过度简化]
“attention 是二次复杂度”不代表整个 Transformer 的所有计算都是 $T^2$；MLP、embedding 和输出层还有不同成本。但它足以解释为什么过长序列是严重问题。
:::

### 4. 压缩率

Lecture 1 定义：

$$
\text{compression ratio} = \frac{\text{UTF-8 byte 数}}{\text{token 数}}
$$

在同一段文本上，值越大，说明平均一个 token 承载的 bytes 越多、序列越短。

```python
def compression_ratio(text: str, ids: list[int]) -> float:
    return len(text.encode("utf-8")) / len(ids)
```

:::caution[比较必须使用同一数据分布]
英文语料上压缩率好的 tokenizer，不一定适合中文、代码或数学文本。报告单个字符串的数字只能帮助调试，不能代表真实 workload。
:::

### Character tokenizer：按 Unicode code point

Python 的 `ord()` 把字符映射到 Unicode code point，`chr()` 执行逆操作：

```python
class CharacterTokenizer:
    def encode(self, text: str) -> list[int]:
        return [ord(char) for char in text]

    def decode(self, ids: list[int]) -> str:
        return "".join(chr(index) for index in ids)
```

优点是概念简单。但 Unicode 包含大量 code point，许多字符极少出现；模型需要很大的稀疏词表，却仍然无法得到特别短的序列。这是“大词表 + 低压缩”的不利组合。

### Byte tokenizer：按 UTF-8 byte

UTF-8 把 Unicode string 映射为 bytes。ASCII 字符通常占一个 byte，中文字符和 emoji 通常占多个 bytes。

```python
class ByteTokenizer:
    def encode(self, text: str) -> list[int]:
        return list(text.encode("utf-8"))

    def decode(self, ids: list[int]) -> str:
        return bytes(ids).decode("utf-8")
```

它的词表固定为 256，任何 UTF-8 文本都能表示，也没有 unknown word。但一个 token 只承载一个 byte，所以压缩率严格为 1；序列对当前 Transformer 架构来说太长。

:::tip[给 SDE 的类比]
Byte tokenizer 像把所有网络协议都暴露成逐 byte 操作：接口极其通用，却把大量低层工作推给了后续计算。
:::

### Word tokenizer：按单词或规则片段

传统 NLP 常用正则表达式把文本拆成单词和标点。它的 token 对人类更有意义，序列也短，但会遇到：

- 训练语料里的独特单词数量可能巨大。
- 拼写变化、屈折变化、URL、代码标识符会制造稀有词。
- 没见过的词需要 `<UNK>`，信息会丢失。
- 词表大小难以精确控制。
- “word” 在中文等语言中没有统一的空格边界。

### BPE：让数据选择常见片段

BPE 从 byte token 开始，反复合并训练语料中最常见的相邻 token pair。常见片段逐渐变成一个 token；稀有片段仍可退化为多个 byte token。

| 目标 | BPE 的处理方式 |
| --- | --- |
| 保证覆盖 | 基础词表包含 256 个 byte 值 |
| 控制词表大小 | 训练固定次数的 merge |
| 缩短常见文本 | 高频相邻片段合并为单个 token |
| 表示新词 | 使用已知子词或 byte 组合，不需要 `<UNK>` |

:::note[BPE 的一句话版本]
BPE 是一种从训练语料学出来的、可控词表大小的 byte 序列压缩启发式。它不理解语义，只统计相邻 token pair。
:::

## Part 5：手工训练一个 BPE

### 训练状态包含什么

从 bytes 开始时：

- `vocab[id] -> bytes`：初始包含 `0..255`。
- `merges[(left_id, right_id)] -> new_id`：记录 merge 的顺序与新 token ID。
- `indices`：训练语料当前的 token ID 表示。

每执行一次 merge，词表通常增加一个 token。因此忽略 special token 时：

$$
V = 256 + \text{num_merges}
$$

### 一轮训练做四步

<Steps>
1. 扫描当前 `indices`，统计所有相邻 pair 的出现次数。
2. 按确定性的 tie-breaking 规则选择最高频 pair。
3. 分配新的 token ID，并把两个旧 token 的 bytes 拼接进 vocabulary。
4. 从左到右把该 pair 的非重叠出现替换为新 ID。
</Steps>

官方 trace 使用 `"the cat in the hat"` 和三轮 merge。为方便阅读，先用字符显示 token，而不是 byte ID：

| 状态 | 当前表示 | 高频候选 |
| --- | --- | --- |
| 初始 | `t h e · c a t · i n · t h e · h a t` | `t+h` 与 `h+e` 都重复出现 |
| merge 1 | `th e · c a t · i n · th e · h a t` | `th+e` 重复出现 |
| merge 2 | `the · c a t · i n · the · h a t` | 后续取决于完整计数与 tie-break |

中点 `·` 只是为了显示空格；真实实现统计的是整数 ID pair。

:::caution[Tie-breaking 是可复现性要求]
多个 pair 次数相同时，必须使用 assignment 规定的稳定规则。依赖 hash map、并行遍历或语言版本的偶然顺序，会让 merge 表和所有后续 token ID 发生分叉。
:::

### 最小训练伪代码

下面表达算法，不是 Assignment 1 的高性能答案：

```python
vocab = {i: bytes([i]) for i in range(256)}
merges = {}
ids = list(training_text.encode("utf-8"))

for step in range(num_merges):
    counts = count_adjacent_pairs(ids)
    pair = choose_pair_deterministically(counts)
    new_id = 256 + step

    merges[pair] = new_id
    vocab[new_id] = vocab[pair[0]] + vocab[pair[1]]
    ids = merge_non_overlapping(ids, pair, new_id)
```

### 非重叠 merge

假设当前 ID 是 `[A, A, A]`，要合并 `(A, A)`。从左向右做非重叠替换，结果应为 `[AA, A]`，不是同时让两个 pair 都参与 merge。

```text
位置:     0   1   2
输入:     A   A   A
          └───┘       ← 消耗位置 0 和 1
输出:       AA    A
```

这个边界很适合单独写 unit test。

## Part 6：训练完成后怎样 encode

训练 tokenizer 与使用 tokenizer 是两个阶段：

```text
训练语料 ── train_bpe ──> vocab + ordered merges

新文本   ── encode ─────> token IDs
token IDs ─ decode ─────> 原文本
```

编码新文本时，先转成 UTF-8 byte IDs，再按照**训练时学到的 merge 优先级**应用相关 merge。不能对新文本重新统计最高频 pair；否则每段输入都会产生不同词表，模型无法共享 embedding。

解码则更直接：

```python
def decode(ids: list[int], vocab: dict[int, bytes]) -> str:
    data = b"".join(vocab[index] for index in ids)
    return data.decode("utf-8")
```

:::caution[不要把 new ID 当成 byte]
ID `256` 以上通常代表多个 bytes 的合并结果。decode 必须查 `vocab[id]` 得到 bytes，不能直接调用 `bytes(ids)`。
:::

## Part 7：从课堂版本到可用实现

官方课堂代码刻意保持简单。Assignment 1 会要求你处理真正决定正确性和速度的部分。

### Pre-tokenization

真实 tokenizer 通常先用正则把大文本拆成片段，再在片段内部统计和应用 BPE。它可以控制跨空格、标点或特定边界的 merge，并使统计更容易并行。

Pre-tokenization 不是 BPE 本身，但会显著改变最终词表。相同语料、相同 merge 数，不同 pre-tokenizer 会得到不同结果。

### Special tokens

例如 `<|endoftext|>` 不是普通字符序列，而是模型协议的一部分。实现需要：

- 检测并原子化保存允许的 special token。
- 防止普通 BPE merge 穿过其边界。
- 明确遇到不允许的 special token 时是报错还是按普通文本处理。

:::tip[给 SDE 的类比]
Special token 更像协议控制帧，而不是用户 payload。把它按普通文本拆开，相当于丢失了控制语义。
:::

### 性能

课堂版 encoding 对每条文本遍历所有 merge，再反复扫描完整序列，容易变成非常慢的实现。训练阶段如果每轮都从零重数所有 pair，也会产生大量重复工作。

优化前先测量并回答：

- 时间花在读取语料、pre-tokenization、pair counting 还是 merge replacement？
- 中间结构是否复制了完整序列？
- 多进程分块后，计数合并是否确定且正确？
- 优化版本是否与小规模 reference implementation 输出完全一致？

本站不会给出可直接提交的优化实现；建议先保留一个慢但明显正确的 reference version，用 differential test 验证快速版本。

## 常见误区

### “BPE 在找单词或词根”

不是。它只合并当前表示中高频的相邻 pair。有些 token 恰好对应单词或词缀，是统计结果，不是语言学规则。

### “词表越大，压缩率越高，所以越好”

不完整。更大词表扩大 embedding 与输出层，稀有 token 也更难得到充分训练。目标是在模型成本、序列长度与数据覆盖之间取平衡。

### “Byte-level BPE 的 token 都是合法 UTF-8 字符串”

不是。单个 token 可以对应某个多 byte 字符的一部分；只有把完整 token 序列拼回 bytes 后，整体才通常能解码成原字符串。

### “Tokenizer 只是无关紧要的预处理”

Tokenizer 决定模型看到的序列长度、词表参数量、多语言与代码的相对计算成本、special token 协议，以及字符边界。它会影响训练和推理的效率与行为。

## 本讲小结

- CS336 的核心视角是：在固定资源下最大化模型质量，效率贯穿 tokenizer、架构、系统、数据与 scaling。
- Tokenizer 是 string/bytes 与有限整数序列之间的协议层，不是简单的 `split()`。
- Character tokenizer 词表大且压缩差；byte tokenizer 词表小但序列长；word tokenizer 覆盖和词表控制困难。
- BPE 从 bytes 开始，用固定次数的高频 pair merge 学习可控大小的词表。
- 大词表会缩短序列，却扩大模型的 embedding/输出层；设计始终是取舍。
- 真正可用的 BPE 还需要确定性 tie-break、pre-tokenization、special token、并行统计与性能优化。
