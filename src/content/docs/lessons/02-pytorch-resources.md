## 这节解决什么问题

“代码可以运行”不等于“资源上可行”。训练前要能估算一层需要多少计算、参数与激活内存，并判断瓶颈更可能来自计算还是数据搬运。

当前主线位置：tokenization → **Transformer 的张量计算与成本** → loss → 参数更新。

## 四本账

1. **参数量**：模型长期保存的可训练数字数量。
2. **激活量**：前向计算产生、反向传播可能需要保留的中间结果。
3. **FLOPs**：浮点运算次数的近似量级。
4. **数据移动**：CPU/GPU 或不同内存层级间传输的字节数。

:::tip[给 SDE 的类比]
FLOPs 像业务逻辑工作量，显存带宽像数据库/网络 I/O。优化计算很多但搬运很少的算子，与优化搬运很多但计算很少的算子，方法完全不同。
:::

## Arithmetic intensity

Arithmetic intensity 大致是“完成的浮点运算 / 搬运的字节数”。值高更可能受计算吞吐限制；值低更可能受内存带宽限制。

## 阅读张量程序的固定动作

- 每行写出输入和输出 shape。
- 标出广播发生在哪里。
- 区分 view/transpose 与真实数据复制。
- 估算最大中间 tensor。
- 只有测量后才下性能结论。

## 理解检查

1. 参数内存和激活内存为什么要分开算？
2. 两个 FLOPs 相同的实现，运行时间为什么可能不同？
3. batch size 增加时，哪些内存项会增加？
